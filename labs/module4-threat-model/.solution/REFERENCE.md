# 参考答案 — 登录服务威胁模型

> 示范用,不唯一。重点是方法到位、落到这个系统上。

## 1. 数据流与信任边界

```
┌─────────┐   用户名/密码    ╎   ┌──────────────┐   SQL 查询   ┌──────────┐
│ 用户    │ ───────────────→ ╎ → │ 登录进程     │ ──────────→ │ users 库 │
│(外部)  │ ←─────────────── ╎ ← │ (app.py)     │ ←────────── │(口令等) │
└─────────┘   登录结果/角色  ╎   └──────────────┘   结果集     └──────────┘
                          信任边界↑
            (网络/用户输入 → 服务端:跨界数据必须校验)
```
- 外部实体:用户(不可信)。
- 进程:登录后端。数据存储:users 表(含口令、角色)。
- **信任边界**:用户输入进入后端处;后端↔数据库之间(SQL 文本)也是一条关键边界。

## 2. STRIDE 威胁清单

| 类别 | 本系统的具体威胁 |
|---|---|
| **S** 假冒 | SQL 注入绕过认证,无凭证即冒充 admin(`admin' -- `) |
| **T** 篡改 | 注入 `UPDATE`/`;DROP` 篡改或删除用户数据;改 role 提权 |
| **R** 抵赖 | 无登录/失败日志,攻击者操作后无法追溯、可抵赖 |
| **I** 信息泄露 | UNION 注入读出口令列;口令**明文**存储,一旦泄露全暴露 |
| **D** 拒绝服务 | 构造昂贵查询或海量请求拖垮数据库;无限流/无锁定 |
| **E** 提权 | 注入使 `role='admin'` 命中,普通用户 → 管理员 |

## 3. 缓解 ↔ 设计原则

| 威胁 | 缓解 | 对应原则 |
|---|---|---|
| 注入(S/T/I/E) | **参数化查询**(数据与代码分离) | 完全仲裁 / 机制经济性 |
| 提权 | DB 账号只读最小权限;角色校验每次进行 | 最小权限 / 完全仲裁 |
| 口令泄露 | 加盐哈希存储(bcrypt/argon2),不存明文 | 失败安全默认 |
| 抵赖 | 记录登录成功/失败审计日志 | (可问责性) |
| DoS | 失败次数限制、限流、查询超时 | 最小公共机制 |
| 默认放行 | 默认拒绝,白名单 | 失败安全默认 |

## 4. 最严重的设计缺陷

**把"用户输入"和"SQL 代码"混在一起拼接** —— 违反了"数据与代码分离 / 完全仲裁"。
它一处缺陷同时引爆 S/T/I/E 四类威胁(绕过认证、篡改、泄露、提权),
且修复成本极低(参数化查询)。其次是**口令明文存储**(放大了泄露后果)。

> 启示:很多"漏洞"本质是**设计阶段**的问题。威胁建模能在写代码前就发现"输入跨信任边界未做隔离"这种根因。
