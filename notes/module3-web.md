# 模块3:Web 安全

Web 安全覆盖浏览器安全模型、服务端输入处理、会话管理和常见注入类漏洞。这里抓重点:
四类常见 Web 漏洞各是什么、根因是什么、怎么防,并配一个 SQL 注入挑战。

---

## 0. 背景:浏览器安全模型

- **同源策略(SOP)**:不同源(协议+域名+端口)的脚本默认不能互读数据,是 Web 隔离的基石。
- 服务端信任不了客户端:Cookie、表单、URL 参数都可被篡改,**一切输入都不可信**。

## 1. SQL 注入(SQLi)

把用户输入**拼接**进 SQL 语句,导致输入被当作 SQL 代码执行。

- 危害:绕过认证、读/改/删整库、有时进一步拿服务器。
- 例:`name='%s'` 里传 `admin' -- ` → 注释掉密码校验。
- **防御**:**参数化查询/预处理语句**(数据与代码分离)是根治法;辅以最小权限账号、输入校验、ORM。
- 动手:`labs/module3-sqli`(登录绕过 + 修复)。

## 2. XSS(跨站脚本)

把用户输入**未转义**地输出到页面,浏览器把它当 HTML/JS 执行 → 攻击者脚本在受害者浏览器里跑
(偷 Cookie、冒充操作)。

| 类型 | 注入点 |
|---|---|
| 反射型 | 输入即时回显在响应里(如搜索词) |
| 存储型 | 输入存库后展示给其他用户(如评论)——危害最大 |
| DOM 型 | 前端 JS 直接把输入写进 DOM |

**防御**:输出时按上下文**转义/编码**(HTML/属性/JS);**CSP**(内容安全策略)限制可执行脚本来源;
框架自动转义;敏感 Cookie 加 `HttpOnly`(JS 读不到)。

## 3. CSRF(跨站请求伪造)

诱导**已登录**用户的浏览器,向目标站点发出非自愿的请求(浏览器会自动带上 Cookie)。
例:用户登录着银行,点了恶意页面里的隐藏表单 → 以用户身份转账。

**防御**:**CSRF token**(请求需带服务器下发、攻击者拿不到的随机值);
Cookie 设 `SameSite=Lax/Strict`(跨站请求不自动带 Cookie);校验 `Origin`/`Referer`。

## 4. 会话劫持(Session Hijacking)

偷到会话标识(session id / token)就能冒充用户。途径:XSS 偷 Cookie、网络嗅探、会话固定。

**防御**:Cookie 加 `Secure`(只走 HTTPS)+ `HttpOnly` + `SameSite`;全站 HTTPS;
登录后**重置** session id;设置过期与空闲超时。

---

## 速查

| 漏洞 | 根因 | 根治防御 |
|---|---|---|
| SQLi | 输入拼进 SQL 代码 | 参数化查询 |
| XSS | 输入未转义就当 HTML/JS | 输出编码 + CSP |
| CSRF | 浏览器自动带凭证 | CSRF token + SameSite |
| 会话劫持 | 会话标识泄露 | Secure/HttpOnly/HTTPS + 重置 session |

> 共同主线:**不可信输入 + 数据被当成代码/凭证被滥用**——和模块1/2 的内存漏洞同根:别信输入、把数据和代码分开。
