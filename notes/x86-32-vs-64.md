# x86 32 位 vs x86-64:漏洞利用视角的区别

> 做 64 位实验前先读这份。这里只讲**原理**;具体某关的偏移/地址要你自己测。

学完你应能回答:同一个栈溢出,从 32 位搬到 64 位,exploit 要改哪几处、为什么。

---

## 1. 寄存器与字长

| | 32 位 (i386) | 64 位 (x86-64) |
|---|---|---|
| 通用寄存器 | 8 个(eax, ebx, ecx, edx, esi, edi, ebp, esp) | 16 个(rax…rsp + **r8–r15**) |
| 寄存器宽度 | 32 位 | 64 位(低 32 位仍叫 eax…) |
| 指针/地址 | **4 字节** | **8 字节** |
| 栈指针 / 基址指针 | esp / ebp | rsp / rbp |
| 指令指针 | eip | rip |

**对 exploit 的直接影响**:写返回地址时,
- 32 位:`struct.pack("<I", addr)`(4 字节),pwntools `p32()`
- 64 位:`struct.pack("<Q", addr)`(8 字节),pwntools `p64()`

字节序两者都是**小端**,不变。

---

## 2. 栈帧布局 → 偏移会变

函数序言都是"保存基址指针":

```
32 位:  push ebp;  mov ebp, esp        # 压入的 ebp 占 4 字节
64 位:  push rbp;  mov rbp, rsp        # 压入的 rbp 占 8 字节
```

栈上从局部缓冲区到返回地址,中间隔着 **saved base pointer**:

```
   [ buf ............ ]
   [ saved ebp/rbp   ]   ← 32位4字节 / 64位8字节
   [ 返回地址        ]   ← 32位4字节 / 64位8字节
```

所以**同一份源码,64 位的偏移通常比 32 位大**(saved 指针多 4 字节,有时还有对齐填充差异)。
→ 结论:**换架构必须重新测偏移**,别套用旧值。

---

## 3. 调用约定:参数怎么传(最重要的区别)

### 32 位(Linux cdecl):参数全压栈
```
push arg2
push arg1
call func          # 返回地址压栈,func 用 [esp+4],[esp+8] 取参数
```

### 64 位(System V AMD64):前 6 个整数/指针参数走寄存器
```
顺序:rdi, rsi, rdx, rcx, r8, r9   (第7个起才压栈)
返回值都在 rax
mov edi, arg1
mov esi, arg2
call func
```

**这条对利用方式影响巨大:**

- **ret2win(跳一个无参函数)**:两者几乎一样,只是地址宽度不同。✅ 简单
- **调用带参数的函数**(如 `system("/bin/sh")`):
  - **32 位**:参数就在栈上,payload 直接排好即可——
    `[system地址][返回地址][ "/bin/sh"指针 ]`,非常直观。
  - **64 位**:参数必须先进 `rdi`,栈上排数据没用。需要借助 **ROP gadget**,
    典型是 `pop rdi ; ret`:先把 `"/bin/sh"` 地址 pop 进 rdi,再跳 `system`——
    `[pop_rdi_ret][ "/bin/sh"地址 ][system地址]`。
  → 这就是"64 位 ret2libc 比 32 位难"的根本原因。

---

## 4. 栈对齐(64 位特有的坑)

64 位 System V ABI 规定:执行 `call` 的那一刻,**`rsp` 必须 16 字节对齐**
(等价:进入被调函数第一条指令时 `rsp % 16 == 8`)。

glibc 里不少函数用 SSE 指令 `movaps`,它要求操作数 16 字节对齐;
**对齐不对会直接崩**,而且崩溃地址在 **libc 内部**,看着像"跳错了",很迷惑。

- `system` 内部几乎**必然**触发(`do_system` 用 movaps)。
- `printf`/`puts` 视具体路径,有时触发有时不。

**修复**:在目标地址前垫一个**裸 `ret` gadget**(`objdump -d bin | grep -m1 'c3 *ret'`),
它多 pop 一次、让 rsp 多偏 8 字节补齐对齐:

```
[ ret_gadget ][ 目标函数地址 ]   # 把进入目标时的 rsp 对齐补回来
```

**32 位没有这个硬性 16 字节要求**,所以新手从 32 位入门更顺。

---

## 5. 地址里的空字节(64 位常见)

64 位地址高位常是 0,例如 `0x00000000004004a6` —— 写成 8 字节含多个 `\x00`。
如果漏洞是 `strcpy`/`gets` 这类**遇 `\0` 截断**的字符串拷贝,空字节会把 payload 截断。

- 用 `read()`(按长度读,不看 `\0`)就没事——本系列实验正是用 `read`。
- 用字符串函数时,这是 64 位要额外注意的点。

---

## 6. ASLR 强度

地址位数越多,随机化熵越高:

- **32 位**:可随机化位很少,ASLR 常可**暴力爆破**。
- **64 位**:熵高,纯爆破不现实,通常得先**泄露地址**(info leak)再算基址。

---

## 小结:32→64 改 exploit 的清单

1. 地址 4 字节 → 8 字节(`<I`→`<Q` / `p32`→`p64`)。
2. **重测偏移**(saved 指针 4→8)。
3. 跳无参函数(ret2win):基本只改上面两点。
4. 调用带参函数:从"栈排参数"改为"**ROP 把参数装进 rdi 等寄存器**"。
5. 留意**栈对齐**(必要时垫 `ret` gadget)。
6. 留意地址里的**空字节**(字符串类漏洞)。
