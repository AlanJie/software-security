# 模块6:实践与案例

把前面学的串起来,看几个**真实世界的著名漏洞**:它们属于哪类、根因是什么、影响多大、怎么修。
读案例的正确姿势:**漏洞类别 → 根因 → 影响 → 修复 → 它印证了哪条原则**。

---

## 1. Heartbleed(CVE-2014-0160,2014)

- **类别**:缓冲区**过读**(out-of-bounds read / 信息泄露)。
- **根因**:OpenSSL 的 TLS 心跳扩展里,服务**信任请求方声明的 payload 长度**,
  取出该长度后直接 `memcpy` 回显,却没核对"实际收到的数据有多长"。
- **影响**:攻击者声称"我的心跳数据 64KB"但只发几字节,服务就回 64KB**进程内存**——
  私钥、会话 cookie、用户明文等被一并带出;可反复抓取。波及全球海量服务器。
- **修复**:校验声明长度 ≤ 实际记录长度,否则丢弃。
- 动手复刻:`labs/module6-heartbleed`(payload 紧邻 secret,length 取大即过读出 secret)。

## 2. Shellshock(CVE-2014-6271,2014)

- **类别**:命令注入 / 输入解析缺陷。
- **根因**:Bash 把环境变量里"看起来像函数定义"的内容**继续解析并执行**其后的命令。
  很多场景(如 CGI)会把用户可控数据放进环境变量 → 远程命令执行。
- **影响**:Web 服务器、DHCP 等大量入口可被远程 RCE。
- **修复**:Bash 不再自动解析函数定义式的环境变量。
- **教训**:**过度信任输入** + 把数据当代码解析(与 SQLi/格式化字符串同源)。

## 3. Stagefright(CVE-2015-1538 等,2015)

- **类别**:整数溢出 → 堆缓冲区溢出(内存破坏)。
- **根因**:Android 媒体库解析 MP4 等媒体元数据时,长度/尺寸计算发生**整数溢出**,
  导致分配过小、随后越界写。
- **影响**:一条彩信(MMS)即可触发,无需用户交互;影响近十亿设备。
- **修复**:加固整数运算与边界检查;系统层面引入更多隔离。
- **教训**:**整数溢出**常是内存破坏的前置;解析不可信的复杂格式风险极高(正适合 fuzzing,见模块5)。

---

## 横向对照

| 案例 | 漏洞类别 | 共同根因 |
|---|---|---|
| Heartbleed | 缓冲区过读 / 信息泄露 | 信任外部长度、缺边界检查 |
| Shellshock | 命令注入 | 过度信任输入、数据当代码 |
| Stagefright | 整数溢出 → 堆溢出 | 不可信解析、整数/边界处理疏忽 |

**贯穿本路线的主线**:几乎所有案例都能归到两句话——
1. **不要信任外部输入**(长度、格式、内容);
2. **把数据和代码/控制信息分开**,并对一切读写做**边界检查**。

> 这些案例也说明:复杂的解析器(媒体、协议、加密)是漏洞重灾区,而 **fuzzing(模块5)**
> 正是工业界自动挖掘这类漏洞的主力,也是自动化漏洞发现实践的重要落点。
